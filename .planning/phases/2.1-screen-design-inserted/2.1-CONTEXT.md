<decisions>
## Implementation Decisions

### Design sequencing
- Follow phase order: DSGN-01 (landing pages + nav) first since Phase 3 implements landing pages next
- DSGN order: landing pages → auth → test flow → dashboard → payments → component sheet
- All 5 landing pages designed in one Pencil session (/, /for-candidates, /for-companies, /pricing, /contact) — they're a system with shared section patterns
- Component reference sheet (DSGN-06) designed last as a summary of what was created, not a foundation
- One .pen file per flow: `landing-pages.pen`, `auth.pen`, `test-flow.pen`, `dashboard.pen`, `payments.pen`, `components.pen`

### Fidelity level
- High-fidelity: final brand colors, Inter typography, realistic content
- English content only — FR is handled by useTranslation() in code, no FR frames needed
- Data-heavy screens (dashboard candidate table, test questions) use realistic seeded data (e.g., "Marie Dupont — Score 78, Product Manager") — not generic placeholders
- Dynamic/generated content (webcam preview, gauge animation, confetti) uses labeled placeholder blocks with notes (e.g., "[Webcam preview]", "[Gauge animation 0→final, 1.5s]") — not AI-generated images

### Mobile frame coverage
- Mobile frames for key screens with substantially different mobile layouts only — not every screen
- Screens that get dedicated mobile frames (375px):
  - Nav: hamburger sheet variant (already built in Phase 2 but spec the open state)
  - Test engine (/test/[id]/questions): full-screen mobile frame (no nav, full viewport — this is a fundamentally different layout)
  - Dashboard: bottom tab bar + candidate card stack (56px tab bar, brand-navy bg, 5 icons)
  - Dashboard mobile views: /candidates/[id], /compare, /talent-pool, /build-test get mobile frames alongside desktop
- Screens that don't need separate mobile frames: auth forms, pricing, checkout, home page sections (handled via Tailwind breakpoints, behavior defined in REQUIREMENTS.md)
- Mobile breakpoint target: 375px (iPhone 14 standard)

### Annotation depth
- State variants: critical states only
  - Form error state: inline field errors (red text below field) + disabled CTA
  - Loading/skeleton state: shown in context for data-heavy views (dashboard table, talent pool grid)
  - Empty dashboard state: candidate table empty + talent pool empty (uses EmptyState component from Phase 2)
- Animation annotations: key non-obvious interactions only
  - Score gauge: 0→final, 1.5s ease-out
  - Calculating screen: staggered step sequence ("Analyzing responses." → "Checking consistency." → "Computing sub-scores." → "Generating CrismaScore.")
  - Confetti trigger: canvas-confetti fires if score > 70
  - Annual billing toggle: price updates live (e.g., $49 → $39/mo)
  - Standard hover/transition behavior not annotated — defined by brand tokens in code
- Component callouts: label reusable Phase 2 components directly on frames (e.g., "← NavShell", "← InviteModal", "← Skeleton", "← EmptyState") so Phase 3+ implementers know what to import

### Claude's Discretion
- Exact layout composition within each Pencil frame (placement, proportion, whitespace)
- Style guide / color palette selection from Pencil's built-in style guides
- Spacing rhythm within brand token constraints
- Exact copy for sections not yet defined (trust bar logos, FAQ items — Phase 3 context locks these; use Phase 3 CONTEXT.md decisions for content)

</decisions>
