## Summary

Phase 2.1 is a pure design phase: no code is written, only `.pen` design files are created in `/design/`. The output is 6 Pencil files covering all 25+ screens that Phases 3–6 will implement. The planner must decompose this work into 6 sequenced tasks — one per `.pen` file — so each DSGN requirement maps to an independently executable design session.

The critical insight for planning: **Pencil requires the desktop application to be running** for MCP operations. All `mcp__pencil__*` tool calls operate against the live Pencil canvas. The CLI (`pencil --agent-config`) can batch-launch design tasks but **cannot create .pen files** — empty `.pen` files must exist first, or Pencil creates them on first open. The planner should sequence a Wave 0 setup task that creates the `/design/` directory and initializes the 6 empty `.pen` files, then runs design tasks per file.

The second critical insight: **brand variables must be imported into each .pen file before designing**. The existing `globals.css` defines all brand tokens (`--color-brand-primary: #1B4FD8`, etc.). Pencil's variable system imports from CSS, so each design task starts with a `set_variables` call to seed the brand palette before frames are drawn.

**Primary recommendation:** Organize Phase 2.1 into 7 tasks: (0) directory setup + file init, (1) landing-pages.pen, (2) auth.pen, (3) test-flow.pen, (4) dashboard.pen, (5) payments.pen, (6) components.pen. Each task follows the same inner pattern: set variables → batch_design frames → get_screenshot to verify → annotate.

## Architecture Patterns

### Recommended Project Structure

```
design/
├── landing-pages.pen    # DSGN-01 — 5 landing pages + nav states
├── auth.pen             # DSGN-02 — sign-up, login, forgot-password + error states
├── test-flow.pen        # DSGN-03 — intro, check, questions (6 formats), calculating, result
├── dashboard.pen        # DSGN-04 — all 5 dashboard views desktop + mobile
├── payments.pen         # DSGN-05 — checkout + success
└── components.pen       # DSGN-06 — shared component reference sheet
```

Files live at project root in `/design/`, not inside `src/` or `.planning/`. This mirrors the CONTEXT.md decision and aligns with Pencil's recommendation to keep `.pen` files alongside the project.

### Pattern 1: Per-file Token Bootstrap

Every `.pen` file must start with a variable import from `globals.css` before any frames are drawn. Pencil variables are not shared across files — each file is independent.

```
Operation sequence per file:
1. mcp__pencil__open_document  → open/create the .pen file
2. mcp__pencil__set_variables  → inject brand tokens from globals.css
3. mcp__pencil__get_variables  → verify tokens loaded
4. begin batch_design operations
```

**Brand tokens to inject** (from confirmed `globals.css` values):

| Token name | Hex value | Use |
|------------|-----------|-----|
| brand-primary | #1B4FD8 | Primary buttons, links, active states |
| brand-secondary | #3B6FE8 | Secondary interactive elements |
| brand-navy | #0F2A6B | Sidebar bg, hero bg, dark surfaces |
| brand-light | #EEF2FF | Page backgrounds, card tints |
| brand-accent | #6366F1 | Accent badges, highlights |
| Inter | (system font) | Body and UI text throughout |
| JetBrains Mono | (system font) | Timer, numeric data, code elements |

### Pattern 2: Frame-Per-Screen

One Pencil frame per screen (or per distinct state). Frame naming convention drives implementer reference:

```
Frame naming pattern:
  [route-slug]              → desktop default
  [route-slug]--mobile      → mobile variant (375px)
  [route-slug]--[state]     → state variant (e.g., "login--error")

Examples:
  "home"                    → / desktop
  "dashboard--mobile"       → /dashboard at 375px
  "login--error"            → /login form error state
  "questions--qcm"          → /questions with QCM format visible
```

### Pattern 3: Component Reuse via ref Nodes

Pencil supports `ref` node type for component instances — changes to the main component update all instances. Mark all reusable Phase 2 components as reusable nodes early so they can be ref'd across frames within the same file.

**Phase 2 components to model as Pencil reusable components** (per file):

- NavShell (desktop glass bar + hamburger mobile) — in landing-pages.pen, reused in auth.pen and test-flow.pen
- Toast (success/warning/error variants) — modeled in components.pen, referenced in landing-pages.pen (contact form success state)
- Skeleton shimmer block — modeled in components.pen, referenced in dashboard.pen
- EmptyState card — modeled in components.pen, referenced in dashboard.pen
- Modal shell (Dialog desktop / Drawer mobile) — modeled in components.pen

**Cross-file component sharing:** Pencil `.pen` format supports an `imports` array — each value is a path to another `.pen` file from which variables and reusable components are available. This means `components.pen` can be imported into `dashboard.pen` to reuse Skeleton and EmptyState components without rebuilding them.

**Import pattern:**
```
dashboard.pen imports: ["./components.pen"]
```
This gives dashboard.pen access to all reusable components defined in components.pen.

### Pattern 4: Annotation via Note Nodes

Pencil has a `note` node type specifically for annotations. Use notes (not text layers inside frames) for:
- Animation specs (e.g., "← slideUp, 200ms, staggerChildren 0.1s")
- Component callouts (e.g., "← InviteModal from src/components/modals/InviteModal.tsx")
- Placeholder labels (e.g., "[Gauge animation 0→final, 1.5s ease-out]")
- Timer color change annotation (e.g., "JetBrains Mono — amber at 1min, red at 30s")

### Design Task Inner Loop (per frame)

```
For each screen frame:
1. batch_design (insert): create frame at named position
2. batch_design (insert): add layout containers (header, main, footer sections)
3. batch_design (insert): add UI elements (text, shapes, ref instances)
4. batch_design (update): apply brand token variables to fills/strokes
5. get_screenshot: verify visual output
6. batch_design (insert): add note annotations
7. snapshot_layout: check for positioning issues (complex frames only)
```

### Anti-Patterns to Avoid

- **Designing mobile before desktop:** Mobile variants are derivatives — always establish desktop frame first, then create the mobile frame as a copy/adaptation
- **Inline token values instead of variables:** Never hardcode `#1B4FD8` as a raw fill — always reference the brand-primary variable. This enables one-pass global updates
- **Over-annotating standard behavior:** Only annotate the 4 specified non-obvious animations; hover states, transitions, and responsive breakpoints are defined in code, not in Pencil
- **Putting components.pen content in other files first:** The component reference sheet is built last, as a summary. Do not build toast/modal variants from scratch in dashboard.pen — callout their existence as annotations instead
- **Treating French content as needed:** Zero FR frames. All localisation is in code via `useTranslation()`

## Common Pitfalls

### Pitfall 1: Pencil Desktop Not Running When MCP Calls Execute

**What goes wrong:** `mcp__pencil__batch_design` and all other Pencil MCP tools fail with a connection error if the Pencil desktop application is not open.
**Why it happens:** The MCP server runs as part of the Pencil desktop process — it is not a standalone server.
**How to avoid:** The task instructions must start with an explicit prerequisite: "Ensure Pencil desktop app is open before executing any mcp__pencil__* tool calls."
**Warning signs:** Any `mcp__pencil__*` tool returns a connection/timeout error on first call.

### Pitfall 2: Variables Not Persisted Across Sessions (Auto-save Absent)

**What goes wrong:** Brand token variables set with `set_variables` in a prior session may not persist if the `.pen` file was not saved before closing Pencil.
**Why it happens:** Pencil documentation explicitly notes "Auto-save is not yet available." Files must be manually saved.
**How to avoid:** Each design task ends with an explicit save instruction. Verify with `get_variables` at the start of any continuation task.
**Warning signs:** `get_variables` returns empty or default variables at the start of a session.

### Pitfall 3: Token Override by UI Kit Defaults

**What goes wrong:** Applying the Shadcn UI kit can overwrite the brand token variable values with the kit's default palette.
**Why it happens:** Kit application may re-initialize the variables block.
**How to avoid:** Apply any UI kit reference first, then immediately run `set_variables` with CrismaTest brand tokens to restore them. The set_variables call should always be the last step of initialization.
**Warning signs:** get_screenshot shows grey/neutral buttons instead of brand-primary blue.

### Pitfall 4: Cross-file Component Changes Don't Propagate Automatically

**What goes wrong:** If a NavShell component is updated in landing-pages.pen, those changes do not automatically appear in auth.pen or test-flow.pen.
**Why it happens:** Cross-file imports in Pencil resolve at load time — editing the source file does not push updates to importing files in real time.
**How to avoid:** Finalize shared components (NavShell, especially) in landing-pages.pen before starting auth.pen or test-flow.pen. Treat the component definitions as locked once used elsewhere.
**Warning signs:** Auth.pen shows old nav state after landing-pages.pen NavShell was revised.

### Pitfall 5: Frame Count Explosion (Dashboard Phase)

**What goes wrong:** Dashboard (DSGN-04) requires 5 views × 2 breakpoints = 10 frames, plus state variants (skeleton, empty, error) = potentially 15+ frames. This makes the file unwieldy and hard to navigate.
**How to avoid:** Include state variant frames only for the 3 specified cases (dashboard table skeleton, talent pool skeleton, empty state). Do not create skeleton variants for every dashboard view.
**Warning signs:** Dashboard task takes disproportionately longer than others; frame count exceeds 15.

### Pitfall 6: Questions Screen Under-speccing All 6 Question Formats

**What goes wrong:** DSGN-03 requires all six question format layouts (QCM, drag & drop, case study, simulation, audio/video, short text) visible. Designing only 1–2 formats and noting "similar for others" leaves implementers without a spec.
**Why it happens:** The /questions screen is complex — it's easy to design the frame container once and under-document the format variations.
**How to avoid:** In test-flow.pen, create 6 distinct QA format panels — either as separate sub-frames within the questions frame or as 6 separate frames named `questions--qcm`, `questions--dragdrop`, etc.
**Warning signs:** get_screenshot shows a single question format without clear indication of how other formats differ.

## Code Examples

Confirmed patterns from official Pencil documentation:

### Variable Definition (Per-file Brand Token Bootstrap)

The `set_variables` MCP call injects brand tokens as document-level design variables. Each variable becomes a CSS-custom-property-equivalent available throughout the file:

```
mcp__pencil__set_variables({
  filePath: "design/landing-pages.pen",
  variables: {
    "brand-primary": "#1B4FD8",
    "brand-secondary": "#3B6FE8",
    "brand-navy": "#0F2A6B",
    "brand-light": "#EEF2FF",
    "brand-accent": "#6366F1",
    "font-sans": "Inter",
    "font-mono": "JetBrains Mono"
  }
})
```

### Frame Creation via batch_design

```
mcp__pencil__batch_design({
  filePath: "design/landing-pages.pen",
  operations: [
    {
      type: "insert",
      node: {
        id: "home-desktop",
        type: "frame",
        label: "Home — Desktop (1440px)",
        width: 1440,
        layout: "vertical",
        gap: 0,
        fill: { type: "solid", color: "var(brand-light)" }
      }
    }
  ]
})
```

### Component Marking (reusable: true)

Per `.pen` format spec — mark a node as reusable to enable `ref` instances elsewhere:

```json
{
  "id": "nav-shell",
  "type": "frame",
  "label": "NavShell",
  "reusable": true,
  ...
}
```

### Cross-file Import Declaration

Per `.pen` format spec — imports array at document root:

```json
{
  "imports": ["./components.pen"],
  "nodes": [...]
}
```

### Note Annotation Node

## Open Questions

1. **Exact `batch_design` operation schema for complex nodes**
   - What we know: `batch_design` supports insert, copy, update, replace, move, delete operations; nodes follow the .pen JSON format (type, id, fill, layout, gap, etc.)
   - What's unclear: The exact parameter envelope wrapping the node JSON for each operation type is not officially documented in detail — only inferred from `.pen` format spec + community sources
   - Recommendation: Plan tasks to iterate — start with simple frame + container structure, use `get_screenshot` to verify, then refine. Do not plan tasks that assume first-attempt perfection

2. **Whether mcp__pencil__open_document can create a new file or only open existing**
   - What we know: CLI docs say "it currently can't create files" (referring to CLI's `pencil --agent-config`); MCP `open_document` behavior on a non-existent path is undocumented
   - What's unclear: Does `open_document` create the file if it doesn't exist, or error?
   - Recommendation: Wave 0 task should create 6 empty `.pen` files using `Write` tool before any MCP calls, ensuring the files exist at the expected paths

3. **Whether annotation notes appear in `get_screenshot` output**
   - What we know: `note` nodes are a distinct type in the `.pen` format; they are "not unlike HTML elements"
   - What's unclear: Whether `get_screenshot` renders notes visually or omits them
   - Recommendation: Place notes outside frame boundaries or in a dedicated annotation layer so they don't obscure the design screenshot
